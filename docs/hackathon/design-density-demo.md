# Design-density demo

This is the deterministic five-minute path for demonstrating Learning Foundry. It uses only prepared local evidence and the prepared execution adapter. The live Codex adapter is optional and should not be part of the primary recording.

The journey register contains eight stages: Source, Theory, Explain, Explore, Transfer, Approve, Apply, and Revise. Opening the first three records local inspection progress only. The remaining stages complete from durable ledger evidence, so browsing a screen cannot falsely claim participation or approval.

## Before recording

1. Run `pnpm dev` and keep the terminal available only for recovery.
2. Use **Restart prepared journey** in the journey bar and confirm the reset.
3. Keep the workspace on **Understanding** with the first journey step active.
4. Select the prepared adapter in Foundry. Do not enable live Codex for the primary take.
5. Set the Electron window to at least 1440 × 900 and hide notifications, unrelated windows, and private material.

The reset confirmation clears runtime demo evidence and restores the committed prepared seed. Use it only when restarting the take.

## Five-minute script

### 0:00–0:35 — Canonical evidence

Open **Inspect the prepared sources**. Select the design-density talk and point to its normalized fragment and source location. State that sources remain canonical and synthesis is reviewable.

Open **Inspect the Living Theory**. Select “Review for user value,” then briefly show its source context, evidence trail, epistemic kind, and unresolved question.

### 0:35–1:15 — Human explanation

Open **Build the human model**. Show one source claim, one synthesized interpretation, and the exact source inspector. State that the artifact is grounded in the active Learning Foundry project rather than presented in source order.

### 1:15–2:05 — Prediction and micro-world

Open **Predict, observe, and reflect**. Change the configuration to:

- Row spacing: `12 px`
- Hierarchy strength: `80%`
- Visible fields: `7`

Choose **Capacity improves, but scanning becomes harder**, reveal the outcomes, and record the configuration. Then record this reflection:

> More rows became visible, but hierarchy had to carry more of the scanning burden.

Point out that prediction, interaction, and reflection are separate evidence records. Moving a slider before recording does not alter memory.

### 2:05–2:45 — Transfer

Open **Apply the model to a new interface** and submit:

> Keep grouping space and target size, strengthen urgency hierarchy, and remove only fields that do not change the operator's next decision.

Use medium confidence and reasoned support. Return to Understanding or Memory and show that the low-transfer-evidence gap is no longer open while human and agent projections remain distinct.

### 2:45–3:40 — Capability gate and application

Open **Inspect and activate the capability**. Show source coverage, assumptions, operating boundaries, evaluation cases, failures, and the relevant understanding evidence.

Approve with:

> The prepared evaluation and transfer evidence satisfy the gate.

Then activate separately with:

> Activate this approved version for the prepared UI review.

Open **Run the prepared design review**, enter:

> Review the Learning Foundry source queue for useful density.

Run the prepared adapter and show the literate result, adapter identity, timing, capability version, and prompt boundary.

### 3:40–4:45 — Correction and revision

Choose **Correction**, set the theory revision focus to **value density**, and enter:

> Preserve spacing that communicates grouping; reduce only space that carries no meaning.

Append the feedback, propose consolidation, and approve it with:

> The practical correction is traceable and should revise both learning and capability work.

Show the three outputs together: targeted human review, the new theory element linked to its predecessor, and the structured capability revision request. The active capability is not silently replaced or reactivated.

### 4:45–5:00 — Close

Return to Understanding and show the completed journey rail and revised Living Theory. Close with:

> Learning Foundry makes shared understanding inspectable and keeps humans and agents able to participate together as systems evolve.

## Recording checklist

- Record the Electron app, not development tools, for the primary take.
- Use macOS Screenshot (`Shift–Command–5`) for a reliable local screen recording; capture the selected Electron window or a tightly bounded screen region.
- Record at native resolution with the pointer visible and microphone levels checked.
- Make one ten-second test recording to verify text legibility and audio before the full take.
- Keep a second take without narration as fallback B-roll.
- Capture a still of Understanding, the micro-world evidence state, and the approved consolidation result.
- Do not expose credentials, imported private sources, terminal history, or the live execution prompt.

## Recovery checklist

- If a command fails, keep the current screen visible, retry once, then restart the prepared journey.
- If Electron is unavailable, run `pnpm dev:web` and record the same prepared path in the browser.
- If live Codex is unavailable, leave **Prepared** selected. This is the canonical demo path, not a degraded mode.
- If persistence contains earlier activity, use **Restart prepared journey** instead of deleting application files manually.
- Preserve the last successful recording before attempting a live-adapter bonus take.
